Why are Microfrontends usefull

  • framework independent - no Framework lock in
  • each microfrontend has its own state
  • no tight coupling
  • independent deployments
  • no shared codebase or conflicts
  • time to market
  • cross-functional teams
Javascript frameworks are not easily interchangeable as they have diverged in capabilities (large projects become inflexible after a year you realize that a whole feature can be implemented easier using react Each microfrontend has its own state -> immutability each app only knows its own state easily interchangeable - more flexibility Independent deployments - No shared codebase or conflicts Time to market

Cross-functional teams

You build it, you run it

  • full control over their features
  • choose the best framework for your needs
  • vertical sliced teams
Cross functional Teams - You build it you run it Teams have full control over their features - can be deployed and maintained independently Each team can use whatever best suits their needs Vertical sliceses in teams

Microfrontend Architecture approaches

Thin global application layer

  • single framework
  • multiple different frameworks
    • Iframe approach
    • single-spa framework
    • Web Components / Angular Elements

Single Framework

  • Namespace attached state
  • Framework dependent component library
    •
    each app has a namespace attached state, or seperate state One Framework dependent Component Library (Angular Material)

    Multiple Frameworks

    • Webcomponent component library
    • DOM is the API
    • Teams can decide which framework to use
    Iframe approach Single-spa React + Angular Elements Component Library written as Webcomponents (works with any framework) DOM is the API Each team has the choice to use the technology that best fits their needs, flexibility, Angular zone.js problems because of global window object pollution --> multiple versions of Angular are difficult.

    Iframe approach

    • simple
    • include apps in iframes
    • use messages to communicate with parent
      • Window.postMessage
      • Pym.js

    single-spa approach

    • developer by CanopyTax
    • support for many SPA frameworks
    • lazy-load applications for improved initial load time
    developed by canopytax because they wanted to migrate their angularJs (angular 1) application React, Angular, AngularJs, Vue, Preact, ember and so on Single-spa functions as a top level router, that mounts and unmounts the child apps

    Things to consider...

    • application size overhead
      • caching
    • makes sense for big projects
    • complex build & integration
    • not the solution for every problem
    many apps --> Application size (Angular Ivy) overhead Maybe Server-side rendering Caching To battle performance impacts Makes maybe sense for big projects with many developers to split it into vertical slices, but for smaller projects other approaches could be easier/better. overhead of maintaining and creating seuch a microfrontend architecture is maybe not worthwhile No CD/CI infrastructure (build pipeline --> complex build processes / complex integration) Microfrontends are not the solution for every problem
